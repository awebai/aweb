Public Registry, Not a Closed Directory

A shared registry for public agent identity. Not the only one.

{{ .Site.Params.description }} The point of awid.ai is to give the ecosystem one widely reachable place to resolve public identity, while keeping the protocol portable enough that other registries can exist, interoperate, and compete. Domains publish controller and registry authority through `_awid` DNS TXT records.

What it should be The default public registry and reference deployment for the protocol.
What it should not be A mandatory identity owner or the only place a permanent identity can exist.
What we encourage Shared public resolution, domain ownership, protocol compatibility, and self-hosting where it matters.

Market Shape
One common public default, many possible registries

Public identity becomes useful when strangers can resolve it. That argues for a shared default service. It does not argue for a monopoly.

Trust Boundary
Read publicly. Write only with signatures.

The public surface should stay small: anonymous reads for discovery, authenticated writes for mutation, and operator endpoints kept out of the protocol story.

Why a shared service matters

If a public did:aw or DNS-backed address is going to work across companies, wallets, agents, and hosts, there has to be at least one registry that everyone can query without prior coordination.

Why exclusivity would be a mistake

Identity systems become brittle when one vendor becomes the only valid resolver. The protocol should support multiple registries, portable state, and verifiable transitions so users can move.

What the site should signal

awid.ai is the hosted public registry and the compatibility target. It is also a thin service you can run yourself. That balance is the product.

Open Surface

What endpoints should actually be public

Open the endpoints needed for resolution and discovery. Keep the write API explicit and narrow. Do not lead with internal health, ops, or deployment-specific routes on the marketing page.

Anonymous reads make the registry useful. Signed writes keep control with the identity or namespace controller. Registry selection starts at DNS, then the client talks to the authoritative awid service for that namespace.

Public read API

GET
/v1/did/{did_aw}/key Resolve the current signing key for a stable identity.
GET
/v1/did/{did_aw}/addresses List public addresses currently attached to an identity.
GET
/v1/namespaces/{domain} Inspect the registered controller and verification state for a namespace.
GET
/v1/namespaces/{domain}/addresses/{name} Resolve a durable public address such as acme.com/support.

Authenticated write API

POST
/v1/did Register a new stable identity with signed creation evidence.
PUT
/v1/did/{did_aw} Rotate keys or update the published server for an identity.
POST / PUT
/v1/namespaces /v1/namespaces/{domain} Register a namespace or rotate its controller with DNS or verified parent authorization.
POST / PUT / DELETE
/v1/namespaces/{domain}/addresses... Create, update, reassign, and remove public addresses under a controlled namespace.

How to explain it on the site

  • Lead with the trust model, not a feature list.
  • Describe the hosted service as the public default, not the sole authority.
  • Separate public reads from signed writes so the surface feels legible.

What technical buyers care about

  • Can identities survive key rotation and server moves?
  • Can domains stay under owner control rather than platform control?
  • Can they self-host later without abandoning the protocol?

What VCs should understand

  • The network effect is in shared public resolution.
  • The defensibility is not lock-in; it is becoming the credible default and compatibility target.
  • The protocol can stay open while the hosted registry still becomes strategically central.
Recommended Positioning

Encourage a two-layer model

Encourage everyone to treat awid.ai as the common public registry for open internet identity, while also making it normal to run compatible registries for private, sovereign, or regulated environments.

  • Use awid.ai for public discovery and cross-organization interoperability.
  • Use your own domain and your own signing keys for authority.
  • Self-host when policy, geography, or product strategy requires it.
  • Keep protocol docs focused on verifiable state transitions, not implementation internals.

What not to overclaim

Do not claim that permanent identity is impossible without awid.ai. That is too strong and strategically unnecessary. The stronger claim is that a shared public registry is the most useful default, and that awid.ai aims to be that default.
